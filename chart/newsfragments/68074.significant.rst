Helm chart now reconciles the Airflow metadata DB bidirectionally on ``helm upgrade``

The ``migrate-database-job`` is now run as a ``pre-install,pre-upgrade`` hook
(was ``post-install,post-upgrade``) and its default args run a reconciliation
script that picks one of three actions depending on the relationship between
the chart's ``airflowVersion`` and the alembic head currently in the database:

* ``target == current`` — no-op.
* ``target > current`` — ``airflow db migrate`` (existing behaviour).
* ``target < current`` — ``airflow db downgrade --to-version <target>``
  executed inside the still-running api-server pod, so reverse alembic
  scripts that ship only in the source-version image are available.

A ``Role`` + ``RoleBinding`` granting ``pods`` and ``pods/exec`` to the
existing ``migrate-database-job`` ``ServiceAccount`` are now rendered when
``rbac.create`` is true (default). The downgrade branch is the only consumer.

Users who set ``migrateDatabaseJob.command`` or ``migrateDatabaseJob.args``
explicitly are unaffected — those values continue to take precedence.

See https://github.com/apache/airflow/issues/68072.
