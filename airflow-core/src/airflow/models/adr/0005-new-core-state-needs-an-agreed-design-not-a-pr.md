# 5. New core state needs an agreed design first, and must not duplicate existing semantics

Date: 2026-07-20

## Status

Accepted

## Context

The models in this directory are Airflow's public vocabulary. A field on
`DagRun`, a value in the `TaskInstanceState` enum, a new lifecycle behaviour
like "retry the whole run", a new listener hook — each of these is a permanent
addition to what every operator, every provider, every UI surface, and every
downstream tool has to understand. Unlike an implementation detail, it cannot be
walked back in a patch release.

Two failure modes account for most of the closed PRs in this area, and they look
different but have the same root: the design was settled in the PR rather than
before it.

**The semantics already exist under a different name.** Contributors reach for a
new column because they do not know which existing concept already carries the
meaning. A `target_date` field on `DagRun` was proposed to answer "what date is
this run processing" — which is exactly what `data_interval_start` /
`data_interval_end` are for, generated by the timetable, with a defined
half-open `[start, end)` semantic. Adding a parallel field would have created two
sources of truth for the same question, each drifting, each needing serialization,
migration, API, and UI support. The same applies to state: proposals for a new
`CHECKPOINTED` task-instance state and a matching exception were closed once the
accepted task-state work demonstrated the same recovery pattern working through
the existing `failed → up_for_retry` flow, with no new state at all.

**The change is a semantic shift dressed as a feature.** Dag-run-level retries,
task-group retries, and Dag-run-level dependency semantics are not additive
fields; they change what a run *means* and how the scheduler, the UI, and every
downstream consumer interpret it. These arrive as complete PRs with tests and
docs and are still closed, because the cost of reviewing a large implementation
is wasted when the first question — *should Airflow work this way?* — has not
been answered. That question belongs on the dev list or in an AIP, where it can
be settled once instead of relitigated in each PR's review thread.

The project's answer is not "no" in either case. It is "not here, not yet, and
not in this order".

## Decision

- **Before adding a field, state, exception, or listener hook to a core model,
  identify the existing concept that already carries the meaning** — timetables
  and `data_interval` for run timing, `run_type` and `backfill_id` for run
  provenance, `try_number` and the retry flow for recovery, `task_state` for
  resumability — and explain in the PR body why it is genuinely insufficient.
- **A change to what a Dag run or task instance *means* goes to the dev list or
  an AIP first**, and the PR references the resulting agreement. Run-level
  retries, new lifecycle states, new dependency semantics, and new expansion
  models are all in this class.
- **Ship the smallest primitive that unblocks the use case.** If an existing hook
  plus a lookup covers it, that beats a new hook; if a keyword on an existing
  hook covers it, that beats a new state.
- **Do not bundle an unagreed concept with its plumbing.** A field, its migration,
  its serialization, its API exposure, and its UI are reviewable together only once
  the field itself is agreed. Before that, the extra layers make the review longer
  without making the design question any easier to answer.

This ADR governs **columns and enum values on the core models**. The parallel
question for attributes on the *serialized* Dag — including the
`run_on_latest_version` configuration hierarchy — is decided in
`../../serialization/adr/0005`; a change that only touches the serialized document
is judged there, not twice.

## Consequences

- Contributors with a real need wait longer before writing code, and some
  proposals are refused outright. That is the intended trade: an unneeded column
  on `dag_run` is forever, a delayed feature is not.
- Reviewers get to answer the design question before reading an implementation,
  which is the only point at which the answer is cheap.
- The model vocabulary stays small enough that the state machine invariants in
  ADR 1 remain statable.
- Legitimate large features do land — via an AIP — and the AIP becomes the
  reference the review checks against, rather than the review inventing the
  contract as it goes.

A change **violates** this decision when it:

- adds a column to `DagRun`, `TaskInstance`, `DagModel`, or another core model
  **without naming, in the PR body, the existing concept it was checked against**
  (run timing, run provenance, retry position, resumability) and why that concept
  is insufficient;
- adds a value to a state enum, a new `Airflow*` exception in the task lifecycle,
  or a new listener hook without an accepted design that calls for it;
- changes what a Dag run or task group *is* — run-level retries, group-level
  retries, new dependency or expansion semantics — with no linked dev-list thread
  or AIP;
- arrives as a large multi-layer implementation (model + migration + API + UI)
  for a concept that has not been agreed;
- stores a second, derived copy of state that another table or field is already
  authoritative for — a digest or cache column whose only purpose is to avoid
  loading the authoritative value is in scope when the PR states how it is kept
  in sync;
- persists data the metadata database is not the right home for — credentials,
  unbounded or opaque blobs — instead of fixing the layer that produced it.

Reviewer prompts — the judgement behind the first bullet, which the diff states but
does not settle:

- Does an existing field already express this? The author's answer is the artefact
  the bullet requires; whether it is *right* is the reviewer's call, and often turns
  on knowing that `data_interval` means what the proposer calls something else.
- Where was this agreed?

## Evidence

- #67329 — "Add `target_date`, a user-defined processing date for Dag runs":
  declined outright; `data_interval_start` / `data_interval_end`, produced by the
  timetable, already carry exactly that meaning.
- #63907, #61336 — Dag-level automatic retries, twice, from different authors:
  both closed as needing discussion first, because run-level retry is a change to
  what a run means, not a new option.
- #61809 — proof of concept for task-group retries: reviewers judged it a large
  change to what task groups are, and asked for a dev-list discussion before any
  implementation.
- #66402, #66410, #66445 — a proposed `CHECKPOINTED` state, its listener hook,
  and its supervisor plumbing: all three closed once the accepted task-state work
  showed the same recovery working through the existing retry flow with no new
  state, no new exception, and no new hook.
- #48868, #42572 — XCom filtering and a streaming iterable operator: both closed
  pending the lazy-task-expansion AIP, since neither works under the current
  requirement that the scheduler knows the expansion length up front.
- #54511 — defensive handling and an index for concurrent rendered-field inserts:
  closed because the failure had been reported once in the several years the
  feature had existed; core model changes wait for a demonstrated pattern.
