# 6. The app and its generated spec must hold across the declared framework version window

Date: 2026-07-20

## Status

Accepted

## Context

The API server is built on a stack Airflow does not control — FastAPI, Starlette,
Pydantic, and Cadwyn for API versioning — and Airflow deliberately declares a
**wide** dependency range for them so it composes with the rest of a user's
environment. That range is a contract in both directions: the app must run on the
oldest version the metadata declares, and it must not break on the newest one that
range admits.

Two things make this sharper here than elsewhere in the repo. First, the published
OpenAPI spec under `openapi/` is **generated by the installed FastAPI**, so the
committed spec is a function of the framework version the generator ran against.
Regenerating on a newer FastAPI than the declared minimum produces a spec the
declared minimum cannot itself produce — the static check then passes or fails
depending on whose machine ran it, and the checked-in artifact silently encodes a
version floor nobody wrote down. Second, framework upgrades change behaviour that
is invisible in the diff: a request-signature change in a template response, a
dependency-finalisation ordering that interacts with the session middleware, an
alias or root-model serialization path that only misbehaves when Cadwyn's version
migrations are in play.

The related failure is dependency hygiene driven by scanners rather than by
Airflow's own exposure. A CVE announced against an upstream package is not by
itself a reason to raise Airflow's declared minimum: raising it narrows what users
can install, and the question that decides it is whether the vulnerability affects
Airflow's use of the package.

## Decision

Framework compatibility is treated as part of this area's contract:

- **A change must work across the whole declared range**, not only the version on
  the author's machine. When a framework release changes a call signature or
  semantics, handle both forms (or raise the declared minimum deliberately, in the
  same PR, with the reasoning) — do not silently adopt the new one.
- **Regenerate the OpenAPI spec with the version the project declares**, and if
  the regenerated spec requires a newer framework release, **bump the declared
  minimum in the same PR**. A spec regeneration whose output depends on the
  contributor's environment is not ready to merge.
- **A spec diff that appears without a corresponding datamodel or route change is
  a signal, not noise** — investigate which dependency moved before committing it.
- **Do not raise a declared minimum solely because upstream published a CVE.**
  Assess whether Airflow's usage is affected; keep ranges as relaxed as the code
  allows. Where a transitive package needs pinning for a security range, prefer
  the lockfile / override mechanism over narrowing a public dependency
  declaration.
- **Prefer capping and raising the problem upstream over an in-repo workaround**
  when a framework release breaks the app in a way that would need a fragile,
  non-obvious hack to route around.

## Consequences

- The committed spec is reproducible: anyone with the declared dependency set
  regenerates the same file, and the static check means the same thing everywhere.
- Users keep a wide installable range, which is the point of declaring one.
- Compatibility shims for more than one framework version live in the code for a
  while, and are a real, if small, maintenance cost.
- Some upgrades are blocked on upstream rather than solved locally, so the project
  occasionally sits behind the newest release on purpose.

A change **violates** this decision when it:

- adopts a framework API that does not exist in the declared minimum version,
  without also raising that minimum and saying why;
- commits a regenerated `openapi/` spec produced by a framework version outside
  the declared range, or one whose diff is unexplained by the accompanying
  datamodel/route changes;
- raises a declared minimum version purely in response to an upstream CVE
  announcement, without assessing Airflow's exposure;
- routes around a framework regression with a non-obvious hack in app wiring
  instead of capping the dependency and pursuing the fix upstream.

## Evidence

- #55929 — an OpenAPI spec regeneration whose output required a newer FastAPI than
  the declared minimum; a reviewer pointed out the minimum has to be bumped
  alongside, and the change did not land as-is.
- #65561 — a spec-amendment PR chasing a generation diff that appeared and
  disappeared between environments and CI runs; closed once the real cause turned
  out to be elsewhere.
- #59685 — removing an alias from a connection response, closed because the
  then-current FastAPI still mis-serialized aliased and root-model objects when
  Cadwyn's version migrations were active; the fix had to wait for the upstream
  release.
- #56214 — a FastAPI dependency-finalisation workaround, judged a non-obvious hack
  with side effects on the execution API; the agreed direction was to cap the
  dependency and raise the need upstream.
- #64121 — a Starlette 1.0 template-response migration that assumed only the new
  signature, breaking the still-supported older releases in the declared range.
- #64402 — a proposal to raise the Starlette minimum in response to an upstream
  CVE: declined on the principle that minimums are not raised unless the issue
  affects Airflow, and ranges are kept relaxed.
