``task_instance_mutation_hook`` now receives the ``DagRun`` as an optional second argument, so a cluster policy can mutate a task instance based on the run -- for example routing it to a different queue based on ``dag_run.conf``. Existing hooks that only declare ``task_instance`` keep working unchanged; a hook that wants the run declares ``def task_instance_mutation_hook(task_instance, dag_run)``. The ``dag_run`` may be ``None`` during early task-instance construction, so implementations should guard for it, and must read it without opening a new database session (the hook runs inside a scheduler transaction that prohibits committing).
