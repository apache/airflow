Fix cron timetables inferring a data interval that is not strictly earlier than the run time on days a DST transition occurs. A manual run whose cron tick fell in a spring-forward gap got a zero-length interval, which also caused ``next_dagrun_info`` to skip the following scheduled period; on the fall-back side the inferred interval could end after ``run_after``, and ``CronTriggerTimetable`` with ``run_immediately`` could select a tick that had not happened yet.
