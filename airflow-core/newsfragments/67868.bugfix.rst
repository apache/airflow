The ``kwargs`` field of trigger objects returned by the REST API (for example in the ``trigger`` of a task-instance response) no longer exposes the decrypted trigger keyword arguments. Those kwargs can contain credentials a deferred operator hands to its trigger (an API key, a token, …), so the field is now always returned empty, as ``"{}"``. The field is retained in the response schema for backwards compatibility — and is now marked ``deprecated`` there so consumers are nudged off it — and the triggerer still decrypts and uses the real kwargs at runtime; only the API representation is emptied.
