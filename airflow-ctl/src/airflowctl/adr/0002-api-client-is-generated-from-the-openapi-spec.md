# 2. The API client datamodels are generated from the server's OpenAPI spec

Date: 2026-07-19

## Status

Accepted

## Context

`airflowctl`'s request/response types in `api/datamodels/generated.py` and
`api/datamodels/auth_generated.py` are **generated**, not hand-written.
`datamodel-codegen` produces them from the server's committed OpenAPI specs
(`v2-rest-api-generated.yaml` and the simple-auth-manager
`v2-simple-auth-manager-generated.yaml`); the `generate-airflowctl-datamodels`
prek hook runs the generator and `pyproject.toml` pins its input/output under
`[tool.datamodel-codegen]`. Both files carry the `# generated by datamodel-codegen`
header.

Those server specs are themselves generated from the API server's Pydantic models
(see the core-API ADR "The public REST API v2 is a generated, stable contract"),
so the chain is server models → server OpenAPI spec → `airflowctl` datamodels. The
server's API is the single source of truth; the client tracks it. Two failure
modes look harmless in a diff: **hand-editing a generated file** (silently
reverted on the next regeneration, and meanwhile claiming a shape the server never
published) and **consuming a field the spec doesn't have yet** (the client drifts
ahead of the contract and breaks against any real server).

## Decision

The generated API layer is treated as a downstream projection of the server's
OpenAPI spec:

- **`api/datamodels/generated.py` and `auth_generated.py` are never
  hand-edited.** To change a shape, change it on the server, regenerate the
  server spec, then regenerate these datamodels via the
  `generate-airflowctl-datamodels` hook — and commit the regenerated files.
- **The server API change drives the client change, in that order.** Client code
  consumes a field only after it exists in the regenerated spec; a client PR
  that needs a new field references (or waits on) the server PR that adds it.
- **Endpoint access is wrapped in `api/operations.py`, transport in
  `api/client.py`.** New endpoints are added as operation wrappers over the
  generated models; raw `httpx` calls are not scattered through command bodies.

## Consequences

The client stays a faithful, mechanically-derived projection of the published v2
API across releases, and regeneration is a cheap, deterministic hook run rather
than manual transcription. Contributors pay a real ordering cost — a client
feature needing a new field cannot land before the server exposes it — but that is
what prevents the client from drifting away from the contract its users depend on.

A change **violates** this decision when it:

- hand-edits `api/datamodels/generated.py` or `auth_generated.py` instead of
  changing the server model, regenerating the server spec, and regenerating the
  client datamodels;
- adds or consumes a datamodel field that is not present in the regenerated
  server OpenAPI spec (client drifting ahead of the published contract);
- changes the `datamodel-codegen` inputs/outputs in `pyproject.toml` or bypasses
  the `generate-airflowctl-datamodels` hook so the committed datamodels no
  longer match the server specs;
- issues raw `httpx` requests from a command handler instead of routing through
  the generated `api/operations.py` layer.

## Evidence

- #67725 — a server-side field type change; generated clients pick up the precise
  shape by regeneration, not a hand-edit.
- #69502 — a server model gains a `team_name` attribute that flows to the client
  datamodels through the spec.
- #61550 — a new server endpoint parameter the client consumes only after the spec
  exposes it.
- #68702 — server API and client kept in lockstep (partition clear) so they do not
  diverge.
