# 2. The API client datamodels are generated from the server's OpenAPI spec

Date: 2026-07-19

## Status

Accepted

## Context

`airflowctl`'s request/response types live in `api/datamodels/generated.py` and
`api/datamodels/auth_generated.py`. These files are **generated** — not
hand-written. `datamodel-codegen` produces them from the API server's committed
OpenAPI specifications: `generated.py` from
`airflow-core/src/airflow/api_fastapi/core_api/openapi/v2-rest-api-generated.yaml`
and `auth_generated.py` from the simple-auth-manager spec
(`v2-simple-auth-manager-generated.yaml`). The `generate-airflowctl-datamodels`
prek hook runs the generator, and `airflow-ctl/pyproject.toml` pins the codegen
input/output under `[tool.datamodel-codegen]`. Both files carry the
`# generated by datamodel-codegen` header for exactly this reason.

Those server specs are themselves generated from the API server's Pydantic
models (see the core-API ADR "The public REST API v2 is a generated, stable
contract"). So the chain is: server models → server OpenAPI spec → `airflowctl`
datamodels. The client is a downstream consumer of the same published v2
contract as the bundled UI's TypeScript client and the published Python client.
The **server's** API is the single source of truth; the client tracks it.

This makes two failure modes possible, and both look harmless in a diff:

- **Hand-editing a generated file.** Tweaking a field in `generated.py`
  directly is silently reverted the next time the hook regenerates, and in the
  meantime the client claims a shape the server never published.
- **Consuming a field the spec doesn't have yet.** Writing client code against a
  field before the server API change lands (and before the datamodels are
  regenerated from the updated spec) drifts the client ahead of the contract; it
  breaks against any real server.

## Decision

The generated API layer is treated as a downstream projection of the server's
OpenAPI spec:

- **`api/datamodels/generated.py` and `auth_generated.py` are never
  hand-edited.** To change a shape, change it on the server, regenerate the
  server spec, then regenerate these datamodels via the
  `generate-airflowctl-datamodels` hook — and commit the regenerated files.
- **The server API change drives the client change, in that order.** Client code
  consumes a field only after it exists in the regenerated spec; a client PR
  that needs a new field references (or waits on) the server PR that adds it.
- **Endpoint access is wrapped in `api/operations.py`, transport in
  `api/client.py`.** New endpoints are added as operation wrappers over the
  generated models; raw `httpx` calls are not scattered through command bodies.

## Consequences

The client stays a faithful, mechanically-derived projection of the published v2
API: what `airflowctl` sends and parses matches what the server actually serves,
across releases. Regeneration is cheap and deterministic, so keeping in sync is a
hook run, not a manual transcription. Contributors pay a real ordering cost — a
client feature that needs a new field cannot land before the server exposes it —
but that cost is what prevents the client from drifting away from the contract
its users depend on.

A change **violates** this decision when it:

- hand-edits `api/datamodels/generated.py` or `auth_generated.py` instead of
  changing the server model, regenerating the server spec, and regenerating the
  client datamodels;
- adds or consumes a datamodel field that is not present in the regenerated
  server OpenAPI spec (client drifting ahead of the published contract);
- changes the `datamodel-codegen` inputs/outputs in `pyproject.toml` or bypasses
  the `generate-airflowctl-datamodels` hook so the committed datamodels no
  longer match the server specs;
- issues raw `httpx` requests from a command handler instead of routing through
  the generated `api/operations.py` layer.

## Evidence

- #67725 — "Type asset_expression in the REST API so the UI does not cast
  through unknown": a server-side field type change; generated clients
  (including `airflowctl`) pick up the precise shape by regeneration, not a
  hand-edit.
- #69502 — "Add team_name attribute to plugins for multi-team": a server model
  gains an attribute that flows to the client datamodels through the spec.
- #61550 — "Add the option to select bundle version parameter on dag run trigger
  endpoint": a new server endpoint parameter that the client consumes only after
  the spec exposes it.
- #68702 — "Add partition clear support to REST API matching the CLI": server
  API and client kept in lockstep so the two do not diverge.
